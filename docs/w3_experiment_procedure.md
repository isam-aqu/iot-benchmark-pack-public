# W3 Interference Experiment Procedure

This document records the finalized W3 interference procedure. The Wi-Fi, Zigbee, and BLE subsets are complete with the same five-replicate structure. It complements the per-run `manifest.yaml`, `notes.md`, and `docs/run_sheet.csv` entries.

## Objective

Evaluate protocol behavior under controlled 2.4 GHz background traffic generated by `iperf3`. For Wi-Fi, latency is measured as ESP32 MQTT RTT. For Zigbee, latency is measured from Home Assistant state-change timestamp (`ha_time`) to the laptop logger receive timestamp (`pi_rx_time_ns`), median-normalized as in W1. For BLE, latency is measured from ESP32 advertiser timestamp (`t_local_us`) to laptop BLE receive timestamp (`pi_rx_time_ns`), also median-normalized as in W1.

The experiment measures how median latency, variability, tail latency, and reliability change as background load increases.

Primary paper metrics:

- median latency
- MAD latency
- P95 latency
- tail inflation factor = P95 / median
- drop rate

Secondary metrics:

- mean latency
- temporal standard deviation

## Required Setup

RF/channel notes for the completed W3 run set:

- Wi-Fi AP: netis WF2419, firmware `V2.2.36123`.
- The AP was later found to use Auto channel selection, so the AP channel used
  for `R600`--`R639` is unknown from preserved metadata and may have changed
  after restarts.
- During `R640`--`R659`, the AP was observed on channel 1, channel width
  40 MHz, control sideband Upper.
- Zigbee: channel 20.
- BLE: legacy primary advertising channels 37, 38, and 39 (`2402`, `2426`, and
  `2480 MHz`). The current laptop BLE logger records received advertisements
  but does not expose the per-packet advertising channel.
- Starting with runs `R678` and later, `run_all.sh` records AP RF fields
  in both `run_metadata.json` and `docs/run_sheet.csv`: AP model, firmware,
  channel mode, primary channel, channel width, control sideband, and RF source.

Interpretation note: because the AP channel was not recorded for `R600`--`R639`,
those Wi-Fi and Zigbee W3 runs should be interpreted as shared-environment
background-load tests, not channel-specific coexistence tests. For `R640`--`R659`,
the observed channel-1 40 MHz upper-sideband setting likely exposes BLE
advertising channel 37 and advertising channel 38, so the completed BLE W3 set
is not a narrow 20 MHz single-advertising-channel overlap case.

For BLE AP-channel extension runs, set the AP to 20 MHz width before running
the sweep so each AP channel targets one BLE advertising channel:
channel 1 -> BLE adv 37, channel 6 -> BLE adv 38, and channel 13 -> BLE adv 39.
Record the actual AP channel, width, and sideband in the run notes. Do not use
channel 13 with 40 MHz upper sideband for the clean BLE sweep; if the AP forces
a different width or sideband, preserve the run but mark it explicitly.

Automatic AP RF capture is partial. A Wi-Fi client associated with the AP can
usually report the current primary channel/frequency and sometimes channel
width (`iw dev wlan0 link` on Linux, or macOS Wi-Fi diagnostics/scan tools), but
the logger laptop Wi-Fi should remain disabled during experiments. Use a
separate Wi-Fi probe if automatic verification is needed. The Netis AP UI is the
most reliable source for width and sideband unless a stable router API is added.

Before every W3 run:

- Home Assistant OS and MQTT broker are running.
- Logging laptop is connected by Ethernet.
- Logging laptop Wi-Fi is disabled.
- Dedicated `iperf3` server Raspberry Pi is reachable at `<private-ip>`.
- No stale logger, ACK helper, BLE logger, Zigbee trigger driver, serial monitor, or iperf process is running.

For Wi-Fi runs:

- ESP32 Wi-Fi node is flashed with W3 firmware `v0.4.0`.
- ESP32 serial port is connected, typically `/dev/serial-port`.

For BLE runs:

- ESP32 BLE beacon node is flashed with W3 BLE firmware `v0.4.1`.
- ESP32 serial port is connected for reset and completion-log capture, typically `/dev/serial-port`.
- BLE logger can see the `ble01` advertiser before the formal run.
- BLE W3 advertisements use experiment code `W3BL`. `run_all.sh` infers this for `W3_ble_auto*` run IDs; when running the BLE logger manually, set `BLE_EXPECTED_EXP_CODE=W3BL`.

For Zigbee runs:

- ZHA is online and `light.benchmark_device` is reachable.
- `home_assistant/automations.yaml` is loaded.
- `input_text.current_experiment_id` and `input_boolean.benchmark_run_enabled` exist.
- A Home Assistant long-lived access token is available as `HA_TOKEN`.
- Pipeline is `0.4.2` or newer so Zigbee MQTT events are deduplicated by Home Assistant `event_id` instead of the fixed Zigbee `seq=0` field.

Check for stale processes:

```bash
ps aux | egrep 'mqtt_logger|mqtt_ack|ble_logger|ha_zigbee_trigger|iperf3|arduino-cli monitor' | grep -v grep
```

Stop stale processes if needed:

```bash
pkill -f mqtt_logger.py || true
pkill -f mqtt_ack.py || true
pkill -f ble_logger.py || true
pkill -f ha_zigbee_trigger.py || true
pkill -f 'arduino-cli monitor' || true
pkill -f iperf3 || true
```

## Zigbee Standard Run Commands

The Zigbee W3 run uses `run_all.sh` for MQTT logging and iperf control, plus `python/ha_zigbee_trigger.py` to drive Home Assistant service calls. The trigger driver:

- sets `input_text.current_experiment_id` to the run scenario ID
- enables `input_boolean.benchmark_run_enabled`
- toggles `light.benchmark_device` for 100 commands at `3000±250 ms`
- writes `<RUN_ID>_zigbee_trigger_schedule.csv`
- disables `input_boolean.benchmark_run_enabled` when it exits

Before running, export the Home Assistant token:

```bash
export HA_URL=http://<private-ip>:8123
```

## Zigbee AP Channel-Overlap Extension (Completed R678-R697)

Purpose: test whether the Zigbee channel-20 stability result still holds when
the Wi-Fi AP is forced onto a 20 MHz channel that overlaps Zigbee channel 20.
Use AP channel 1 as the 20 MHz non-overlap/control block and AP channel 9 as the
overlap block. Keep Zigbee channel 20, the same `light.benchmark_device` entity, the same
100-command jittered schedule, and the same 50 Mbps TCP heavy load.

Before each AP block:

- Set AP channel mode to fixed/manual.
- Set AP channel width to 20 MHz.
- Set AP channel to the block channel.
- Confirm the AP UI shows the intended channel, width, and sideband.
- Pass the AP RF fields to `run_all.sh` with `--ap-*` options.

Completed matrix:

| AP setting | Zigbee exposure | Rep1 | Rep2 | Rep3 | Rep4 | Rep5 |
|---|---|---|---|---|---|---|
| Channel 1, 20 MHz | control/non-overlap for Zigbee ch20 | `R678` baseline, `R679` heavy | `R680` baseline, `R681` heavy | `R682` baseline, `R683` heavy | `R684` baseline, `R685` heavy | `R686` baseline, `R687` heavy |
| Channel 9, 20 MHz | overlaps Zigbee ch20 near 2450 MHz | `R688` baseline, `R689` heavy | `R690` baseline, `R691` heavy | `R692` baseline, `R693` heavy | `R694` baseline, `R695` heavy | `R696` baseline, `R697` heavy |

Dedicated outputs:

- `analysis/w3/reports/w3_zigbee_channel_overlap.md`
- `analysis/w3/tables/w3_zigbee_channel_overlap_runs.csv`
- `analysis/w3/tables/w3_zigbee_channel_overlap_pairs.csv`
- `analysis/w3/tables/w3_zigbee_channel_overlap_summary.csv`

Summary: AP channel 9 overlap with Zigbee channel 20 produced a small positive
mean P95 shift relative to matched baselines (`+0.71 ms`), compared with a small
negative shift in the channel-1 control block (`-0.38 ms`). No missing Zigbee
events occurred in either block. The overlap-minus-control P95 contrast is
approximately `+1.10 ms`, so the extension does not show a large Zigbee
tail-latency degradation under the tested 50 Mbps TCP load.

## Required Per-Run Artifacts

Each formal W3 run should contain:

```text
experiments/runs/<EXP_RUN_ID>/
├── manifest.yaml
├── notes.md
├── raw/
│   ├── <RUN_ID>_mqtt_events.csv
│   ├── <RUN_ID>_power_samples.csv
│   ├── run_metadata.json
│   ├── <RUN_ID>_ble_events.csv         # BLE runs only
│   ├── <RUN_ID>_serial.log             # Wi-Fi and BLE runs
│   ├── <RUN_ID>_zigbee_trigger_schedule.csv  # Zigbee runs only
│   ├── <RUN_ID>_iperf.log              # loaded runs only
│   └── <RUN_ID>_iperf_summary.json     # loaded runs only
└── analysis/
    ├── summary.json
    └── metrics.json
```

Minimum manifest content:

- run ID and scenario ID
- workload, protocol, topology, interference, load level
- comparison group, run type, replicate ID
- firmware, pipeline, HA config, and analysis versions
- actual `run_all.sh` command used for the experiment, omitting secrets
- analysis commands
- quality checks

Minimum notes content:

- purpose
- setup
- deviations from protocol
- ESP32 completion line
- iperf confirmation for loaded runs
- data-quality observations
- interpretation
